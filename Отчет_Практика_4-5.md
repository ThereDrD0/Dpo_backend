# Отчет по практике 4-5

Тема: endpoint routing, параметры маршрутов, dependency injection и custom middleware в ASP.NET Core.

В проекте сделано приложение CampusRouteLab на Minimal API. Данные студентов хранятся в памяти, без базы данных и контроллеров.

Реализованные маршруты:

- `/students` - список всех студентов.
- `/students/{group}` - студенты выбранной группы.
- `/students/{group}/{id}` - один студент по группе и id, если нет студента, возвращается 404.
- `/reports/{section?}` - отчет, если секция не указана, используется `overview`.
- `/portal/{module=home}/{page=index}/{id?}` - маршрут со значениями по умолчанию.
- `/files/{**path}` - catch-all маршрут для вложенного пути.
- `/routes` - список зарегистрированных endpoint-ов.
- `/diag/lifetimes` - проверка Singleton, Scoped и Transient.
- `/diag/lifetimes/check` - проверка, что transient создается заново.
- `/diag/request-services` - получение сервисов через `HttpContext.RequestServices`.
- `/diag/app-services` - получение singleton-сервиса через `app.Services`.

В DI зарегистрированы:

- `IStudentCatalogService` как Singleton.
- `IAppInfoService` как Singleton.
- `IRequestContextService` как Scoped.
- `ITransientMarkerService` как Transient.
- `DiagnosticsReportService` как Scoped.

Также добавлен `RequestAuditMiddleware`. Он пишет простые сообщения для `/diag` маршрутов и добавляет заголовки `X-App-Instance`, `X-Request-Id`, `X-Transient-Id`.

Почему scoped-сервис нельзя передавать в конструктор middleware: middleware создается один раз и живет долго, а scoped-сервис должен создаваться отдельно для каждого HTTP-запроса. Если положить scoped-сервис в конструктор, он фактически станет жить как singleton, и это сломает смысл scoped. Поэтому `IRequestContextService` и `ITransientMarkerService` передаются в `InvokeAsync`, где ASP.NET Core берет их из области текущего запроса.

Проверка: проект собирается командой `dotnet build`, маршруты были проверены через HTTP-запросы на локальном сервере.
